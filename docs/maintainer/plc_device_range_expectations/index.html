    PLC Device Range Expectation Table

    Scope

    • Date: 2026-03-13
    • PLC: Mitsubishi MELSEC iQ-R (R08CPU)
    • Host: 192.168.250.101
    • Source of configured ranges: current PLC device-setting screen provided by the user
    • Library policy: do not hard-code these PLC-setting-dependent limits in the client
    • Current probe inputs:
    • internal_docsrc/iqr_r08cpu/current_plc_boundary_specs_20260313.txt
    • internal_docsrc/iqr_r08cpu/device_range_probe_latest.md

    This document records the current PLC-side configured device ranges and the expected response pattern when a request stays inside the configured range, crosses the upper bound, or starts outside the configured range.

    These values are evidence for the current validated target only. They are not client-side hard limits and must not be generalized to other PLC models or other parameter sets.

    These expectations are for the normal direct-device command family:

    • read: 0401
    • write: 1401

    The table is only applicable when the device family itself is supported by the chosen command path. If a device family is rejected for command/path reasons, a different end code may appear even when the requested number is inside the configured range.

    • 0401 range failure: 00 ff ff 03 00 01 04 02 00
    • 1401 range failure: 00 ff ff 03 00 01 14 02 00

    Current practical interpretation on this PLC:

    • 0x4031: device number range or allocation mismatch for the requested device

    Important Exceptions

    These are not pure range issues. They are command-path or target-specific behavior already observed on the current PLC.

    1. S
    2. S0 read is OK.
    3. S0 write returned 0x4030.
    4. Therefore, S write cannot be interpreted only by configured range.

    5. Current project policy keeps typed S device write unsupported instead of treating this as an open investigation.

    6. LT / LST

    7. Direct LTC, LTS, LSTC, LSTS access returned 0x4030.
    8. Use LTN / LSTN-based helper APIs instead:
    9. read_ltc_states
    10. read_lts_states
    11. read_lstc_states
    12. read_lsts_states
    13. read_long_timer
    14. read_long_retentive_timer
    15. In the current range probe:
    16. LTN1024 out-of-range write returned 0xC051

    17. LSTN1024 out-of-range write returned 0xC051

    18. G / HG

    19. These are not governed by the device-range setting screen above.
    20. Direct 0401 access returned 0xC05B.
    21. Extended Specification 0401/0082 access returned 0xC061.
    22. The currently verified practical path is 0601/1601 via:
    23. cpu_buffer_read_*
    24. cpu_buffer_write_*

    25. extend_unit_*

    26. ZR

    27. The current manual device table and the validated target do not agree on the ZR numbering base.
    28. The validated target behavior is decimal, and the current implementation follows that live result.
    29. The configured upper bound corresponds to ZR163839 in this client.
    30. ZR163840 start is rejected with 0x4031.
    31. However, requests starting at ZR163838 / ZR163839 still succeeded even when the requested span clearly crossed into ZR163840 or later.
    32. Focused live results:
    33. reads: ZR163838 x3, ZR163839 x2, ZR163839 x3, ZR163839 x16, ZR163839 x64 all returned 0x0000
    34. same-value writes: ZR163838 x3, ZR163839 x2, ZR163839 x3, ZR163839 x16, ZR163839 x64 all returned 0x0000
    35. Current practical interpretation for this PLC:
    36. ZR acceptance appears to be checked primarily by start address, not by full end-of-span overrun
    37. this is an observed target behavior, not yet a general protocol rule
    38. a separate capture from another PLC environment showed successful direct 0401/1401 access to ZR1535996, so this validated-target boundary must not be treated as a protocol-wide or client-wide ceiling

    39. that large-ZR target was later revalidated through the current Python client by writing 1234 to ZR1535996 and reading back the same value

    40. R

    41. The current device-setting screenshot did not include R, so the exact configured setting was not taken from the screenshot.
    42. However, live probing detected the last accepted start address as R32767.
    43. Focused comparison showed:
    44. R32767 reads/writes starting there still succeeded for x2, x16, and x64
    45. R32768 reads x1, x2, and x16 all returned 0x4031
    46. Current practical interpretation for this PLC:
    47. R behaves like ZR in that acceptance appears to depend mainly on the start address rather than the full end-of-span overrun

    48. this is an observed target behavior, not yet a general protocol rule

    49. LC

    50. LCN511 2-point in-range read/writeback was OK.
    51. LCN512 out-of-range write returned 0xC051.

    52. So LCN write-side out-of-range behavior differs from the simpler 0x4031 seen on many 1-word families.

    53. Z

    54. Live probe detected the last accepted start address as Z19.
    55. Z19 x1 read/write was 0x0000.
    56. Z19 x2 read and Z20 x1 read/write were 0x4031.

    57. So Z behaved like the simpler D/W/SW family, not like R/ZR.

    58. LZ

    59. Live probe detected the last accepted start address as LZ1.
    60. LZ1 x1 and LZ1 x2 reads were both 0x0000, while LZ2 x1 read was 0x4031.
    61. LZ1 x1 write returned 0xC051, while LZ1 x2 write returned 0x0000.
    62. LZ2 x1 write returned 0xC051, while LZ2 x2 write returned 0x4031.
    63. Current practical interpretation:
    64. LZ write behavior is governed not only by start address but also by the required word-unit count

    65. this matches the manual meaning of C051H

    66. RD

    67. Repeatable focused recheck on 2026-03-13 corrected the earlier ad hoc note for RD.
    68. RD524286 x2 read/write was 0x0000.
    69. RD524287 x2 read/write was 0x4031.
    70. RD524286 x16 and x64 were also 0x4031.
    71. Current practical interpretation:
    72. RD does not currently show the same start-address-based overrun acceptance as R and ZR on this validated PLC
    73. treat RD as a simpler boundary family unless a later recheck shows otherwise

    Client Behavior Policy

    • The Python client must not embed PLC-setting-dependent limits such as "D ends at 10239".
    • The client may validate only protocol-format limits that are independent of PLC settings.
    • Device-range acceptance must be determined by the PLC response for the actual target.
    • Current deliberate project exception: R is treated as capped at 32767.
